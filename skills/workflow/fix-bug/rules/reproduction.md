# Reproduction Lock

Phase 2.5. **Before** holistic analysis (Phase 3). Constructs a deterministic artefact that
reproduces the bug — the executor's `FAIL_TO_PASS` contract.

This rule **does not write tests itself** — it picks the right test layer, then delegates to the
appropriate existing skill. The actual RED-GREEN-REFACTOR cycle, framework discovery, and
project-specific test conventions all live in `/tdd`. Browser-driven flows live in
`/e2e-testing`; mobile flows in `/e2e-testing-mobile`.

Source: SWE-agent ([NeurIPS 2024](https://proceedings.neurips.cc/paper_files/paper/2024/file/5a7c947568c1b1328ccc5230172e1e7c-Paper-Conference.pdf)),
[SWE-bench Verified harness](https://www.swebench.com/SWE-bench/), RepairAgent
([ICSE 2025](https://software-lab.org/publications/icse2025_RepairAgent.pdf)).

## Layer routing

Pick the **lowest** test layer that can capture the bug. E2E is expensive — defer to lower
layers when the same assertion is possible. This mirrors the layer-decision rule used by
`/e2e-testing` ([`skills/testing/e2e-testing/rules/layer-decision.md`](../../../testing/e2e-testing/rules/layer-decision.md)).

| # | Bug shape | Layer | Skill to invoke |
|---|-----------|-------|-----------------|
| 1 | Pure logic / business rule (no DOM, no network) | **Unit** | `Skill("tdd", "<feature/bug description>")` |
| 2 | Single component's render output, prop handling, or interaction | **Component** | `Skill("tdd", "<component behaviour>")` — `/tdd` writes a Vitest/Jest + RTL test |
| 3 | Hook behaviour (useState, useEffect, useReducer firing wrong) | **Component** | `Skill("tdd", "<hook behaviour>")` — `/tdd` writes a `renderHook` test |
| 4 | API / DB seam (handler + real DB, FE + BFF contract) | **Integration** | `Skill("tdd", "<seam behaviour>")` — `/tdd` writes a Vitest test against MSW or a test DB |
| 5 | Multi-page navigation, auth, or real network — web | **E2E (web)** | `Skill("e2e-testing", "<flow>")` |
| 6 | Multi-screen flow on Expo / React Native — mobile | **E2E (mobile)** | `Skill("e2e-testing-mobile", "<flow>")` |
| 7 | Visual-only (colour, alignment, spacing — no behaviour) | **Visual** | See [Visual / pixel layer](#visual--pixel-layer-best-effort) — usually best-effort |
| 8 | Performance regression (frame drops, INP, TBT) | **Profile** | Out of scope here — capture as best-effort, route to `/profile-optimizer` after the fix |

When in doubt, climb **down** the table, not up. A unit test that captures the bug is worth more
than an E2E that captures it.

## Unit + component layer (delegate to /tdd)

For rows 1–4 in the routing table:

```text
Skill("tdd", "<one-paragraph description of the failing behaviour, including the
exact assertion the failing test should make. Reference the Evidence Record's
'Affected code' table for the file under test. Path the test at repro/<id>.test.<ext>
to align with /fix-bug's convention.>")
```

The `/tdd` skill handles:

- Framework discovery (Vitest / Jest / pytest / Go testing / cargo test).
- File naming and directory placement matching project convention.
- The RED phase — confirming the test fails for the right reason before any fix.
- Coordinating with `code-quality` during REFACTOR.

`/fix-bug` consumes only the **RED phase output**: the test path and the failing-on-HEAD command.
GREEN/REFACTOR happen later inside the executor (Phase 6) under `/tdd`'s discipline.

### React / React Native rendering bugs (row 2 nuance)

When the bug is "the component renders the wrong thing" — wrong text, missing element, wrong
className, wrong conditional branch — keep it at the component layer. The `/tdd` test pattern is
React Testing Library + the project's existing render harness:

```tsx
// Generated by /tdd. Asserts the buggy behaviour from the Evidence Record.
test('AlertBanner renders the dismiss button when dismissible is true', () => {
  render(<AlertBanner message="..." dismissible />);
  expect(screen.getByRole('button', { name: /dismiss/i })).toBeInTheDocument();
});
```

Anti-patterns to avoid (also flagged by the [Phase 7 verifier](./independent-verification.md#verifier-checks)):

- **Snapshot tests as the only assertion** — a snapshot freezes implementation, not behaviour.
  The verifier cannot tell whether a snapshot diff is the bug fix or unrelated drift. Use
  explicit assertions on what the user sees (text, role, aria attributes).
- **Testing internal state** — `expect(wrapper.state().count).toBe(1)` couples the test to
  implementation. Test through the rendered output instead.
- **Mocking the component itself** — if the bug is in `<AlertBanner>`, do not mock `AlertBanner`.
  Render it.

For React Native specifically, use `@testing-library/react-native` (or the project's existing
RN test harness). The same routing rule applies: a render bug is component-layer, not E2E. Reach
for `/e2e-testing-mobile` only when the bug requires multi-screen navigation, native modules, or
device-specific behaviour.

### Hook bugs (row 3 nuance)

```tsx
// Generated by /tdd for a useEffect-firing-twice bug.
test('useFetchUser does not refetch when userId is stable', () => {
  const fetcher = jest.fn();
  const { rerender } = renderHook(({ id }) => useFetchUser(id, fetcher), {
    initialProps: { id: '42' },
  });
  rerender({ id: '42' });
  expect(fetcher).toHaveBeenCalledTimes(1);
});
```

## User-flow layer

For rows 5–6, the bug needs real navigation, real auth, or real network. Delegate:

- **Web**: `Skill("e2e-testing", "<flow description>")`. The skill drives Playwright Test Agents
  (Planner / Generator / Healer). Phase 0 of `/e2e-testing` halts and asks before installing
  Playwright if the repo isn't wired up — confirm with the user before saying yes.
- **Mobile (Expo / React Native)**: `Skill("e2e-testing-mobile", "<flow description>")`. The
  skill drives Maestro YAML flows. Phase 0 halts before installing Maestro CLI.

The locator ladder is owned by those skills (role → label → `data-testid` for web; `testID` for
mobile). `/fix-bug` does not propose locators directly — it delegates.

A user-flow repro is **expensive** in tokens and wall-clock time. If the bug can be captured at
the component layer instead, climb down the table.

## Visual / pixel layer (best-effort)

Visual-only bugs (a colour is wrong, a button is misaligned, the layout breaks at a specific
viewport) often cannot be captured by behavioural tests. Two options:

| Option | When to use |
|--------|-------------|
| **Storybook story + visual diff** | The project has Chromatic / Percy / Loki wired up. The story becomes the repro. |
| **Markdown checklist** (`repro/<id>.md`) | No visual-diff tooling — document the manual reproduction steps with screenshots, mark as best-effort |

Neither produces a `FAIL_TO_PASS` contract the [Phase 7 verifier](./independent-verification.md)
can mechanically check. Surface this to the user and document in the bug-notes ledger.

## What counts as a valid repro

Regardless of layer, a repro is **valid** if all four hold:

1. **Runnable** — a single command produces a fail/pass outcome.
2. **Failing on HEAD** — exit code non-zero, with an assertion or output that matches the
   Evidence Record's symptom.
3. **Deterministic** — three consecutive runs produce the same failure. Flakes are not repros —
   they need a stabilisation pass before they can serve as the contract.
4. **Pinned** — same input, same seed, same env. No reliance on wall-clock time or external
   services unless those are mocked.

If any of these fail, fall back to [Best-effort fallback](#best-effort-fallback).

## Path convention

```text
repro/<short-bug-id>.<ext>
```

`<short-bug-id>` examples: `dash0-trace-abc123`, `linear-sup-456`,
`stacktrace-typeerr-userservice`. The `<ext>` matches the project's test convention; `/tdd`
selects it.

For best-effort markdown checklists: `repro/<id>.md`.

## Best-effort fallback

Some bugs cannot be captured deterministically by any layer:

- Race conditions that need specific timing or scheduler state.
- Production-only bugs needing real data, real load, or specific tenant configs.
- Heisenbugs that disappear under instrumentation.

Write a **best-effort** repro: a script or test that gets as close as possible (e.g., a
load-test for perf, a stress harness for races), or a markdown checklist with manual
reproduction steps. Mark it explicitly:

```markdown
### Reproduction
- Path: repro/<id>.md
- Status: best-effort (manual steps; no automatable assertion)
- Reason: race condition needs production scheduler state
```

Do **not** block on this — Phase 3 proceeds. The executor (Phase 6) cannot use it as a
`FAIL_TO_PASS` contract; the [Phase 7 verifier](./independent-verification.md) will skip its
`FAIL_TO_PASS` check and rely on the broader test suite plus diff-sanity instead.

## Forbidden reasons to skip Phase 2.5

Phase 2.5 is **not optional**. The only valid reasons to mark a repro as best-effort are
the three listed in [Best-effort fallback](#best-effort-fallback) (race conditions,
production-only bugs, Heisenbugs) plus rows 7 and 8 of the
[layer routing table](#layer-routing) (visual-only, performance regression). These five
reasons — and only these five — are also the closed list the
[Phase 5 reproduction gate](../SKILL.md#step-5a--reproduction-gate-mechanical) accepts.

The following are **NOT** valid reasons to skip the `/tdd` / `/e2e-testing` /
`/e2e-testing-mobile` delegation. If the agent finds itself reaching for any of these,
it has misclassified the bug shape and must climb **down** the routing table instead —
not sideways into best-effort:

- **"The diff is small."** A two-line fix is exactly the case where a regression test
  is cheap and load-bearing — the fix is one careless edit away from being reverted
  and nothing else in the suite prevents that.
- **"The pattern is already exercised by `<other test>`."** Other tests assert other
  invariants. The new bug needs its own assertion or the regression cannot be detected
  by the broader suite.
- **"A test harness for `<framework>` would be too much scaffolding."** React Testing
  Library, `renderHook`, `QueryClientProvider`, MSW, Vitest, pytest fixtures, and the
  standard mocking libraries are existing project conventions in any non-trivial
  codebase — not new scaffolding the agent has to author. If the project genuinely
  lacks a test harness for the layer the bug lives at, capture that as a separate
  project-onboarding ticket; the agent must not silently absorb it as "best-effort."
- **"The bug is so localised the test would just duplicate the fix."** A test that
  calls the buggy code path and asserts the corrected behaviour is the `FAIL_TO_PASS`
  contract — its job is duplication of intent, not duplication of implementation. The
  fix can change shape later; the assertion stays valid as long as the behaviour does.

The agent's confidence in the fix is not a substitute for the repro. Confidence is
scored at Phase 4 separately from the repro existence check; both must clear before
Phase 5 even sees the proposal, and the [Phase 5 gate](../SKILL.md#step-5a--reproduction-gate-mechanical)
fails-closed (no force-proceed) when the repro is missing.

### Common bugs that trigger this rationalization despite being captureable

If a bug appears in this table, "scaffolding overhead" is not a valid justification —
the test pattern is already established by the listed routing row.

| Bug shape                                                            | Correct routing                                                            |
|----------------------------------------------------------------------|----------------------------------------------------------------------------|
| React Query cache invalidation gap (`onSuccess` did not invalidate)  | Row 3 (hook) — `renderHook` + `QueryClientProvider` + assertion on cached state |
| Missing dispatch / event listener / subscription                     | Row 3 (hook) or row 2 (component) — depending on where the listener lives  |
| Wrong conditional branch in a component                              | Row 2 (component) — RTL `render` + role-based assertion                    |
| Server contract mismatch (wrong key, wrong shape, wrong status code) | Row 4 (integration) — MSW handler returning the buggy contract + assertion |
| Off-by-one in a pure function                                        | Row 1 (unit) — direct function call + boundary-value assertion             |
| Stale closure in a `useEffect` / `useCallback`                       | Row 3 (hook) — `renderHook` + `rerender` with new dependency value         |

The list is non-exhaustive. When the bug shape is not listed, climb the routing table
top-to-bottom and pick the lowest layer that can assert the corrected behaviour.

## Bisect fast-path

Runs only if **all** hold:

- Phase 1 pre-flight produced a regression window (`last_green_sha` set).
- A valid (non-best-effort) repro exists from a layer that can run on a checked-out commit.
- The repository has a working `git bisect` history (no shallow clone, no broken commits in the
  range).

Procedure:

```bash
git bisect start
git bisect bad HEAD
git bisect good <last_green_sha>
git bisect run <repro-command>
```

`<repro-command>` must be the exact command from the Evidence Record's `Reproduction.Command`
field, returning non-zero on failure.

When bisect finishes, capture the offending commit:

```markdown
### Bisect result
- Offending commit: <sha>
- Author: <author>
- Subject: <commit subject>
- Diff size: <N> lines
- Files: <list>
```

If the offending commit's diff is **≤ 50 lines**, route to Phase 5 directly with proposal "revert
or amend `<sha>`". The bug-notes ledger records `hypotheses[0].state = confirmed`.

If the diff is larger, continue to Phase 3 — but pass the offending commit to holistic-analysis as
a strong hint.

Source: [git-bisect docs](https://git-scm.com/docs/git-bisect),
[Gun.io 2025 bisect guide](https://gun.io/news/2025/05/git-bisect-debugging-guide/).
