# autonomous-workflow — Developer Documentation

This file is auto-loaded by Claude Code when working inside
`skills/autonomous-workflow/`. It captures **why this skill exists, how it's
shaped, and what to keep in mind when changing it** — context that an agent
or contributor needs *before* editing the procedures.

For the runtime instructions the skill follows when executing a task, read
[`SKILL.md`](./SKILL.md) and [`rules/*.md`](./rules/). This file is about
**design intent**, not runtime behavior.

---

## What this skill is

`autonomous-workflow` is an **orchestrator skill**. It does not contain the
intelligence to plan, code, test, review, or deliver — it contains the
sequencing and gates that decide *when* those things happen and *which
companion skill* handles each one.

Concretely:

- **Phase model 0 → 7** — every autonomous task moves through the same
  phases in order; each phase has a gate that must pass before the next.
- **Companion skills** — the actual work (planning, TDD, UX review, code
  quality, doc updates, PR creation, CI fixing) is delegated to specialist
  skills in this same repo.
- **Artifacts** — Full Mode produces `plan.md` and `walkthrough.md` in
  `.agent/{branch}/` so a fresh Claude session can pick up where another
  left off.

If you find yourself adding domain knowledge to this skill (how to write a
React component, how to fix a TypeScript error, how to write a good commit
message), you're in the wrong place — that knowledge belongs in a companion
skill (`ux`, `code-quality`, `create-pr`, etc.) which this skill invokes.

## Why this skill exists

LLM agents tend to fail in three predictable ways during long-running tasks:

1. **Skip validation** — start coding before they understand the requirement.
2. **Lose context mid-task** — a session ends, a new one begins, the new
   session has no memory of decisions made.
3. **Burn tokens on hallucinated fixes** — get stuck on a failing test and
   try increasingly speculative fixes for 20+ iterations.

`autonomous-workflow` exists to address all three:

1. **Phase 0 (Validation) is mandatory** — the agent must confirm
   understanding with the user before any code is written.
2. **`plan.md` is the single source of truth** in Full Mode — generated by
   the `create-plan` companion, it captures every decision, requirement,
   and rationale so a new session can resume cold.
3. **Phase 4 stuck-loop has a hard limit of 3 iterations** — after 3 failed
   attempts on the same area, the agent runs `confidence(bug-analysis)` and
   escalates to the user. No more 20-iteration token burns.

Everything else in this skill (companion invocation, mode detection,
artifacts, parallelization) is in service of those three goals.

## The companion-skills pattern

The defining design decision in v3 is that **the workflow is modular by
default**. Companions are not hard dependencies; they're advisory invocations.

### How a companion is wired

A phase rule contains a section like:

```markdown
## TDD Trigger

When the task involves pure logic, business rules, or "test-driven" was
requested, invoke:

    Skill("tdd")

If the skill is unavailable, log one line and continue:

    - [TIMESTAMP] Phase 3: tdd — not available, continuing
```

Three things matter:

1. **The trigger condition is concrete** — file globs, keyword match, file
   counts. Not "if the agent feels appropriate." Subjective triggers leak
   judgment into runtime and make behavior non-reproducible.
2. **The `Skill()` invocation is wrapped in graceful degradation** — if the
   companion isn't installed, the workflow logs a one-liner and continues.
   Never block on a missing companion. **The only exception is `confidence`
   at Phase 1**, which is the load-bearing safety gate.
3. **The section anchor matches the "Disable by" link** in
   `rules/companion-skills.md`. If you rename the heading, update the link
   target — GitHub anchors are slug-derived from headings (with `.` stripped,
   so `## Claude.md Trigger` becomes `#claudemd-trigger` not
   `#claude-md-trigger`).

### Adding a new companion

1. Confirm the companion skill exists in `agent-skills.git/skills/<name>/`.
2. Add a row to the registry table in `rules/companion-skills.md`. Include
   the trigger condition (concrete), args, and the disable link.
3. Add the invocation in the relevant `phase-N-*.md` rule, with an `## Anchor`
   heading matching the disable link target.
4. Verify the workflow still degrades gracefully when the new companion is
   missing (uninstall the companion locally, run a task, confirm it skips).
5. Update the install commands in `SKILL.md`, `README.md`, and `install.sh`
   if the new companion should be in the recommended install set.

### Removing a companion

1. Delete the row from `rules/companion-skills.md`.
2. Delete the invocation block in the relevant phase rule.
3. (Optional) Note in `CHANGELOG.md` so users know.

### Why some companions can't be disabled

`confidence` at Phase 1 is the only non-removable companion. Without it, the
plan gate is gone and the workflow loses its primary safety mechanism. If a
user wants to remove this gate, they should fork the skill and adjust
`phase-1-planning.md` directly — not silently disable the gate by uninstalling
the companion.

## The `.agent/` artifact directory

Artifacts (Full Mode `plan.md` and `walkthrough.md`) live in
`.agent/{branch-name}/` inside the worktree. **Not** in `.gw/` (that's the gw
CLI's territory for its own config). **Not** on the main branch (artifacts are
worktree-scoped — committing them to main creates context confusion).

The directory name `.agent/` (singular) was chosen to align with the
`~/.agents/skills/` cross-tool discovery convention used by Codex, Cursor,
OpenCode, and other Agent Skills–compatible clients. Naming consistency
across the ecosystem is the goal — so other tools can adopt the same artifact
location if they want to.

Earlier versions used `.gw/` because the workflow was tied to gw-tools.git.
That coupling is gone in v3. The `.gw/` reference now appears only in
migration notes and "never use this for artifacts" reminders.

## The 3-iteration stuck-loop limit

The single most important number in this skill is **3**.

Earlier versions of the workflow let agents iterate on a failing test for up
to 20 attempts before stopping. Empirically, the agent rarely converges past
attempt 3 on the same area — when it does, it's almost always wrong (it's
fixing symptoms, not root cause). Continuing past 3 means burning tokens on
hallucinated fixes that the user is going to throw away anyway.

The protocol:

```
After 3 iterations on the same failing area:
  → Skill("confidence", "bug-analysis")
  → Present findings + summary of attempts
  → Ask user: continue / try different approach / stop
```

Phases 4 (testing) is where this fires most often, but the same logic applies
anywhere the agent is iterating on the same problem. The number 3 is
deliberately tight — it's a respect-the-user signal, not a punishment.

When changing this number, update **all** of:

- `rules/phase-4-testing.md` (`## Stuck-Loop Detection`)
- `rules/safety-guardrails.md` (`## Stuck-Loop Limit (Phase 4)`)
- `rules/companion-skills.md` (`## Stuck-Loop Protocol`)
- `rules/error-recovery.md`
- `rules/overview.md`
- `SKILL.md` Core Principles
- `README.md` Migration Note
- `templates/agent.template.md`

Otherwise the limits drift across surfaces and the agent gets contradictory
instructions.

## Mode detection (Full vs. Lite)

Every task starts with mode detection. The decision changes which artifacts
are produced and which companions run.

| Mode     | Criteria                              | Artifacts | Companions                                   |
| -------- | ------------------------------------- | --------- | -------------------------------------------- |
| **Full** | 4+ files OR complex / architectural   | Required  | All applicable                               |
| **Lite** | 1-3 files AND simple / straightforward| None      | Phase 0, Phase 2, Phase 5 update-claude, Phase 6 create-pr only |

When in doubt, choose Full. Lite is for genuinely small changes — not for
"the user said it's small."

Phase 0 and Phase 2 are mandatory regardless of mode. Phase 5 `update-claude`
runs in both modes (the self-improving doc loop is cheap and matters for
small changes too — they still drift docs).

## Parallelization

Three places benefit from sub-agent fan-out:

1. **Phase 1 research** (`Explore` sub-agents per package/concern) — when
   the task is complex, parallel exploration finds context faster than
   sequential reads. See [`rules/phase-1-planning.md#parallel-research`].
2. **Phase 7 CI fixes** (`ci-auto-fix` sub-agents per failed check) — cap
   at 2 handoffs per PR; each sub-agent has its own retry budget. See
   [`rules/phase-7-ci-gate.md#parallel-ci-fixes`] and
   [`../create-pr/SKILL.md`] Step 8.

Phase 3 (implementation) is **not** parallelized. File-level changes share
state (lockfiles, types, imports), and parallel writes create conflicts that
cost more time than serial implementation saves.

## Things to keep invariant

When editing this skill, do not break these — they're load-bearing:

- **Phase 0 and Phase 2 are mandatory.** Validation and worktree isolation
  are not optional.
- **`confidence` at Phase 1 is non-removable.** It's the only companion that
  must always run.
- **Companions degrade silently.** Never make the workflow block on a
  missing companion (except the one above).
- **No AI co-author tags.** Never add `Co-Authored-By` lines to commits or
  PRs generated by this workflow. The user has reasons.
- **Anchors must match disable links.** Every `## Anchor` heading in a phase
  rule that's referenced from `companion-skills.md` must produce a slug
  matching the link. GitHub strips `.` from slugs (so `## Foo.bar` →
  `#foobar`, not `#foo-bar`).
- **Artifact paths are `.agent/{branch}/`, never `.gw/{branch}/`.** The only
  acceptable mention of `.gw/` is a migration note or "never use this"
  reminder.
- **The system-prompt for the agent template stays lean.** It references
  `SKILL.md` for procedures rather than duplicating them. If the agent
  template grows beyond ~250 lines of system prompt, it's drifting.

## Working on the skill

### Editing a phase rule

1. Read `SKILL.md` and `rules/companion-skills.md` first to refresh the
   contracts.
2. Make the change. Keep the rule file self-contained — an agent should be
   able to load this one file plus the two above and execute the phase
   correctly.
3. Verify anchors still match disable links (`grep -E "phase-.*\.md#" rules/companion-skills.md`).
4. If the change affects companion behavior, update `rules/companion-skills.md`
   in the same PR.

### Editing the system prompt (in `templates/agent.template.md`)

The template is what gets symlinked into `~/.claude/agents/autonomous-workflow.md`
by `install.sh`. It's the entry point Claude Code uses when the user says
"implement X independently." Keep it lean — it should reference `SKILL.md`
and `rules/companion-skills.md` rather than duplicate their content.

### Testing changes end-to-end

There's no automated test suite for this skill. The validation pattern is:

1. Symlink the skill locally (`bash install.sh --global` or `--project`).
2. Run a small Lite Mode task — verify Phase 0 + Phase 2 + Phase 5 + Phase 6
   companions all fire correctly.
3. Run a larger Full Mode task — verify `plan.md` and `walkthrough.md` are
   generated, confidence gates fire, stuck-loop respects the 3-iteration cap.

If you change the stuck-loop number, mode-detection criteria, or the
companion list, do both runs.
