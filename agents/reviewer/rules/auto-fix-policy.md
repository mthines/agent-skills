# Auto-fix policy

Auto-fix runs only in `reviewer` (own-work) and only when the conditions below all hold. The split exists because applying fixes under the user's identity to local files is reversible (`git`), but applying fixes that the user didn't ask for is unwelcome.

## When auto-fix runs

| Condition | Source |
| --- | --- |
| Mode is **Fix Mode** | default for `reviewer` with no `--report` and no own-PR detected |
| OR Mode is **Self-Review** | own PR detected via authorship check |
| AND `--report` was NOT passed | explicit user override |
| AND the change is on the user's own branch / own PR | the only target |

If any condition fails, the agent proposes-only.

## Simple issues — fix immediately

| Category | Action |
| --- | --- |
| Unused imports and variables | Remove |
| Lint / formatting errors | Apply the linter's autofix |
| Missing type annotations where the type is obvious from initializer | Add |
| Typos in comments, docs, and string literals | Fix |
| Inconsistent whitespace / indentation | Normalize |
| Dead / commented-out code | Remove |

Note each fix briefly (one line per fix) in the Post-Fix Summary.

## Complex issues — plan only, never apply

A complex fix is anything that:
- Touches a function signature (callers may break).
- Renames a public export.
- Changes test expectations.
- Modifies more than ~10 lines in one file.
- Touches a generated file or migration.
- Removes a feature flag or env-var reference.

For each complex finding, emit:

```
### <issue title>
**Why:** <one-sentence explanation>
**Fix plan:**
1. <step 1>
2. <step 2>
**Files involved:** <list>
```

The user reviews the plan and applies it manually (or invokes another tool).

## Forbidden auto-fix targets

These files / paths are **never** auto-fixed, even for simple-looking issues:

| Path pattern | Reason |
| --- | --- |
| `**/migrations/**`, `**/migrate/**`, `**/prisma/migrations/**` | Once committed, reverting requires a follow-up migration |
| Lock files (`package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, `Cargo.lock`, `go.sum`) | Regenerated by the package manager; manual edits cause drift |
| `dist/`, `build/`, `out/`, generated TypeScript declaration files | Will be regenerated; edits are noise |
| `.env*`, `*.env`, anything in `secrets/` | Sensitive |
| Anything matching `**/*.generated.*` | Author intent is "do not hand-edit" |
| Test snapshots (`__snapshots__/`, `*.snap`) | Regenerated by the test runner |

If a simple-looking finding falls on a forbidden target, propose-only.

## Post-fix summary

After applying all simple fixes:

```
### Auto-fixed
- src/foo.ts:42  — removed unused import `Bar`
- src/baz.ts:101 — fixed typo in error message ("recieve" → "receive")

### Planned (not applied)
- src/qux.ts — rename `runJob` to `executeJob` (touches 7 callers)

### Verification
- Lint: PASS
- Type-check: PASS
- Tests (scoped to changed files): PASS
```

Re-run lint + type-check + scoped tests after auto-fix to confirm no regressions. Report new errors only (ignore pre-existing). On a regression, **revert the offending auto-fix** and downgrade it to "Planned (not applied)" — never leave the working tree in a broken state.

## What this rule does not cover

- Auto-fix on someone else's PR — forbidden in `pr-reviewer` (auto-fix is not even a phase there).
- Auto-fix on the user's own PR when `--report` was passed — propose-only.
- Auto-fix decisions for findings that came from `--with` lenses — lenses produce findings; the auto-fix policy applies uniformly regardless of source.
